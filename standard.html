1. Source Code Style

1.1. Indentation

Tabs, not spaces. Tabs must be eight characters wide.

1.2. Curly Brace Placement

The opening curly brace must be on the same line as the declaration, not on the line below it.

1.3. Line Length

The shorter the line, the better. Line length must never exceed 80 characters.

1.4. Naming Conventions

Everything must be written with snakecase. All names must be in the same case. By default, names must be all lowercase. The only exception is with #define directives, when names must be in all uppercase (SNAKECASE).

All function names in a program must be prefixed with the name of the program, in the format of "<name-of-program>_<function-name>".

All type defintion names must be suffixed with "t". All enum names must be suffixed with "e". All struct names must be suffixed with "s".

1.5. Nesting

Never nest code more than three layers deep. Ever.

1.6. #include Directives

#include directives are to be ordered according to length. The following is an example: 

#include <time.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <some_random_lib.h>

1.7. Pointer declaration

All asterisks used in pointer declaration must have one space between the base datatype and one space between the name of the pointer. If there is no pointer name, such as in a cast, only put a space between the asterisk and the base datatype.

2. Design Conventions

2.1. Inter-function Communication

All user-written functions must follow a convention for return codes. The only exception to this is the main function which must use unix return codes (0 for success, 1 for failure). Every user-written function must return an "int".

The following are the standard return codes: -1: Print an error message, free all allocated memory, then exit the program. 0: False (Function failed) 1: True (Function succeeded)

Any other return codes are defined within a program. These are the only return codes which always mean the same thing.

If a function has data that needs to be returned, a variable must be passed by reference and the data must be stored in that variable.

2.2. Function Responsibility

Every function should only have one responsibility. Don't write functions that do multiple things. Each function should only have one job.

2.3. Text-based Data

If a program is going to be dealing with definitions of data, then implement a way for that data to be read from a plain text format as soon as possible. A good format for this is JSON.
